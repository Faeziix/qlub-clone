Autonomous Milestone Loop

/next-milestone drives the deterministic milestone-cycle workflow — select a milestone → for each issue implement ↔ (code ∥ QA ∥ design review) → decision-gate → open ONE sign-off PR, for Faeziix/qlub-clone.

deterministic JS control flow code reviewer — reads the diff QA agent — runtime/errors (HTTP + logs) design reviewer — drives Chrome + impeccable approved / integrated escalate / pause for human decision

2 · The review fan-out & decision-gate (one round)

Three independent reviewers run in parallel against the same pushed code; the gate is the barrier that merges them. The QA agent stays browser-free so it never contends with the design reviewer's live Chrome.

3 · The boards — who owns what, and when

Critical / High block the gate and drive fix-rounds; Medium / Low are advisory and surfaced in the PR. Each board only runs when it's relevant.

❗ Errors / Crashes QA agent

500s, hydration warnings, console errors, blank renders, failed requests. Browser-free (HTTP + dev-server log). Runs on any runnable-surface issue.

🎨 Styling / Visual design reviewer

Driven by impeccable (critique/detect/audit): layout, spacing, responsive framing, design-system consistency, anti-"AI-slop". UI issues only.

🧩 Business-Logic / Flow design reviewer

Dead-end / orphan routes, missing buttons (e.g. no pay entry), can't re-order, cart not clearing, wrong totals. UI issues only.

🌐 RTL / i18n conditional

Untranslated strings, wrong dir, digit/number formatting, LTR flash, wrong default locale. Only when the change touches an i18n surface.

What makes it trustworthy

  • three lenses A code reviewer (diff), a QA agent (runtime errors via HTTP + logs), and a design reviewer (Chrome + impeccable) run in parallel — each catches a class of bug the others miss.
  • no contention The QA agent is browser-free (HTTP + dev-server log), so it never fights the design reviewer for the one connected Chrome.
  • design-aware The styling board is driven by impeccable (critique/detect/audit), not eyeballing — scored against your PRODUCT.md/DESIGN.md when present.
  • severity-gated Only critical/high findings block; medium/low ride along as advisory polish, so the gate stays fast.
  • relevance-gated The design pass self-skips non-UI issues, the QA pass skips issues with no runnable surface, and the RTL/i18n board only runs when i18n is touched — no tokens burned where there's nothing to check.
  • bounded while (round < MAX_ROUNDS) caps implement↔review at 5 passes (raised from 3 for the extra reviewers) — no infinite ping-pong.
  • graceful If a reviewer can't run (no browser, app won't boot), it skips (non-blocking) and the others remain the gate.
  • one sign-off A whole milestone lands as a single PR; escalations (secrets, legal, irreversible) pause the loop for you instead of guessing.
  • live on GitHub Labels and the Project board (#2 "Qlub Iran — Delivery") move per-issue as it happens — claim → In Progress, integrate → Done — not stamped in bulk at start/PR time. The Revision round and Reviewers Project fields track each issue's progress.
  • audit trail The implementer and all three reviewers each comment on the issue (🤖 / 🔎 / 🧪 / 🎨), so the issue thread tells the whole story without opening the diff.
  • cost-aware Reviews are scoped to each issue's own base..head delta (not the accumulating branch — avoids O(n²) re-reads), green lenses are skipped on fix rounds, tests run once (code lens only), and Opus reviewers run at medium effort. Relevance-gating means a docs-only change pays for one reviewer, not three.
